Introduction

Rostas is the coordinator app for managing Church & Community volunteer rosters. It currently connects to the Infoodle database to access volunteer data, and handles notifications, reminders, and SMS replacements automatically using the Twilio SMS service. Both these services are paid services requiring a subscription. 

This guide covers everything you need: from collecting availability through to handling replacements. Part A introduces Rostas with a video and overview. Part B covers Roster Coordinator tasks. Part C covers additional Overall Coordinator tasks for those who have database access. Part D covers expert/technical tasks requiring Apps Script access.

Part B Roster Coordinators Guide 

1. Getting Started with Rostas

Rostas is built primarily for desktop access, though it can also be used on a tablet. 

The Rostas Views


When you open Rostas you will see these views in the left-hand sidebar. Tap the relevant view to navigate to that area of the system.

The Run Commands view is your main working area. This is where you trigger all roster actions — sending emails, building the roster, and sending notifications. Each button runs a single action and shows you the result when it is done. After tapping a button, Rostas queues the action and runs it within a few minutes. A Complete message appears in the Status column when done — always check this before moving on.


Reading Results

Every action writes a result message in the Command history panel. Common results:

  • Filled: 12, Pending: 3, Unfilled: 0 — roster built successfully. Pending means a volunteer marked Unsure but has been provisionally assigned.
  • Unfilled: 2 — two slots could not be filled. Review the roster and handle manually.
  • Sent to: 14 volunteers — notification emails dispatched.
  • Error: … — something went wrong. Note the message and contact tech support if it repeats.

The Roster Cycle

Each roster follows the same sequence of steps. Run them in order — skipping a step or running them out of order can produce unexpected results.

3. Step 0 — Refresh the Seasons Calendar

Run this at the start of each roster build and any time you make changes to season dates. It recalculates all roster dates, overlays NZ public holidays and school holidays, and updates the calendar view.

Step 0 — Refresh calendar

In Rostas → Run Commands, tap 0. Refresh calendar. Wait for the result message to confirm completion. Check the Seasons Calendar view to confirm dates look correct.

⚠️ Always run Step 0 after editing season start/end dates, adding a new season, or at the beginning of a new roster cycle.


Step 2 — Role Preferences

Role preferences tell the system which roles each volunteer is willing to do. The system needs this information before it can build the roster.

Option A — Volunteers Set Their Own Preferences

Volunteers receive a personalised link and tap through each role, choosing 'Yes I do this', 'Happy to fill in', or 'Not for me'.

Step 2 — Send role preference emails

In Rostas → Run Commands, tap 2. Send role preference emails. Volunteers receive an email with a personalised link. Allow a few days for responses before building the roster. Check who has responded in the Volunteers Roles view.  


Option B — You Set Preferences for the Volunteer

If a volunteer is not comfortable using the link, or for new volunteers who have not yet set preferences, you can set their roles directly in Rostas.

Open the Volunteers Roles view. Select the volunteer on the left, then review and edit their role assignments on the right.

💡 When to use each option: The Roles List view shows all active roles and their settings. The Volunteers Roles view shows per-volunteer assignments. Send the email link at the start of a new season; set preferences directly in Rostas for day-to-day adjustments or less tech-confident volunteers.  For some rosters coordinators may prefer to set roles themselves. 


Volunteer Frequency Settings

Frequency settings control how many times per month a volunteer can be rostered for a given roster type. The roster engine strictly enforces these caps.

To set or update frequency:

  1. In Rostas, go to the Individual Details view. Select the volunteer from the left panel.
  2. Use the arrow buttons (‹ ›) to switch between roster types if the volunteer serves in more than one roster.
  3. In the Max Days Rostered panel (bottom centre), tap 1, 2, 3, or 4 to set the monthly maximum. The default is 4 for weekly rosters, 2 for fortnightly rosters.
  4. Add a note if helpful (e.g. '4th Monday only' or 'family commitments 1st Wednesday'). 
  5. Save. The change takes effect at the next roster build.
⚠️ Frequency caps are enforced automatically at roster build time. A volunteer who has already reached their monthly cap will not be assigned to any further dates in that month, even if they are marked Available.


6. Step 3 — Send Availability Emails

Availability emails go to all active volunteers for your roster type. Each volunteer receives a personalised link showing the upcoming dates. They tap each date to mark themselves Available, Unsure, or Can't make it, then tap Save.

Step 3 — Send availability emails

In Rostas → Run Commands, tap 3. Send availability emails. The result confirms how many emails were sent. Allow 5–7 days for volunteers to respond. Volunteers who do not respond are treated as Unavailable at roster build time.

The Three Availability States
✅ Available — The default starting state for all dates.
❓ Unsure — Signals caution. The volunteer may be provisionally assigned.
❌ Can't make it — Immediately removes them from the scheduling pool for that date.
Everything starts as Available — volunteers only spend time changing the dates they cannot make.

What if someone missed the email?

You can update a volunteer's availability directly in Rostas via the Individual Details view. Select the volunteer from the left panel, then use the Availability Requests panel (bottom right) to find the date and update their status.

Step 4 — Build the Roster

The roster engine assigns volunteers to roles based on their availability and role preferences. It works through four priority tiers and within each tier selects the volunteer who has been rostered least recently.

Step 4 — Build roster

In Rostas → Run Commands, tap 4. Build roster. The Music roster can take up to 10 minutes — this is normal. The system builds the next 8 weeks of dates by default (configurable per roster type in Settings — ROSTER_BUILD_WEEKS_{Type}). Check the result message: Filled, Pending, and Unfilled counts. Review the roster in the Global Roster view before sending.


Reviewing the Result — The Traffic Light System

🟢 Filled
Volunteer confirmed and ready. No action needed.
🟡 Pending
Provisionally assigned (volunteer marked Unsure). Monitor.
🔴 Unfilled
No suitable volunteer was found. Attempt to fill before sending.
⚠️ Do not send notifications until you have reviewed the roster. Check for Unfilled slots and any obvious problems first. Once notifications are sent, volunteers receive their assignments immediately.



Editing the Roster Before Sending

In the Global Roster view, tap or click any assignment to open the edit panel on the right. You will see three options: Take particular notice of any red (unfilled dates - These dates may require you to find a replacement or it may be workable to have less volunteers on duty that week)


You also have the Following Options at this Stage

  • Replace Volunteer — a drop-down offers a list of possible replacement volunteers.
  • Save as Filled — manually confirm a Pending (Unsure) assignment.
  • Save as Declined — this automatically starts the SMS replacement chain to find a substitute.


Step 5 — Preview Communications

Before sending notifications, use the Preview button to see exactly what the notification email and SMS will look like. This is the same view volunteers will see.

Step 5 — Preview communications

In Rostas → Run Commands, tap 5. Preview communications. A browser page opens showing the notification email template. Check volunteer names, roles, dates, and links all look correct. You can also adjust the wording of SMS messages here or in the Settings view. Close the preview and proceed to Step 6 when satisfied.

💡 Keep any wording changes generic and appropriate for all roster types, or revert changes afterwards so the next coordinator does not encounter problems.


Step 6 — Send Notifications

Notifications go to every volunteer with a Filled or Pending assignment. Each volunteer receives an email with their role and date, and a link to confirm (for pending) or decline. An SMS is also sent.

Step 6 — Send notifications

In Rostas → Run Commands, tap 6. Send Notifications. The result confirms how many volunteers were notified. Volunteers receive their email and SMS within a few minutes.

📬 What volunteers receive: Each volunteer gets an email showing their name, role, and dates rostered. The email includes a Can't make it link. If they tap it, the SMS replacement chain runs automatically — coordinators do not need to do anything when this happens. There is also a link to a simplified printable roster. For Pending (Unsure) assignments, volunteers are shown Yes, I can make it and Can't make it buttons so they can confirm or decline a date they were unsure about when the availability email was sent.


Manual Replacement

Use manual replacement when a volunteer contacts you directly — for example, they phone to say they cannot make it and suggest someone else, or you need to make a swap yourself without waiting for the SMS chain.

Manual replace

Go to Rostas → Global Roster or Individual Details view. Select the assignment you want to change. Tap Replace Volunteer, then select the replacement from the dropdown. Save. The system processes the swap within 1 minute. Check the Status column — it changes from Pending to Done when complete.

📬 The replacement volunteer receives an automatic notification after a manual swap, with the usual links to the "I can't make it" button, roster view, and print roster.


11. The SMS Replacement Chain

When a volunteer taps 'Can't make it' in their notification email, the system automatically begins contacting other available volunteers by SMS one at a time. 

What happens automatically:

  • The system finds the next suitable available volunteer for that role and date.
  • It sends them an SMS with a link to accept or decline.
  • If they accept → they are added to the roster and the chain stops.
  • If they decline or do not respond within 2 hours (standard) or 15 minutes (urgent/imminent events) → the next volunteer is contacted.
  • If no one accepts → the coordinator receives an email letting you know the slot is still unfilled.
🌙 Quiet Hours: SMS messages are not sent between 9:00 PM and 7:00 AM NZ time. If the chain is running overnight, it pauses and resumes in the morning. Quiet hours can be adjusted in Settings — search for 'Quiet'.
🔄 Fairness — Being replaced has one coded consequence: When a volunteer is replaced — whether they tapped Can't make it or were manually swapped by a coordinator — the system remembers. The next time a replacement is needed for a role they can do, they are given gentle priority to be contacted first. This gives them the chance to step up when they're free. They can always decline, no explanation needed. Once they serve in any capacity, this priority is automatically cleared. There's no ongoing obligation — just a quiet nudge to give willing volunteers a chance to contribute again.


12. Checking the Status Dashboard

The Status Dashboard shows the status of all notifications sent from Rostas — emails, SMS reminders, and replacement requests. Check it after running any notification step to confirm delivery. If an error appears, use the Email Tech Support button to send an error report to Tech Support that automatically contains the error report.

Note: the full Audit Log is accessible to Technical Support only. If something does not look right, contact Tech Support and send the error message.

Common things to look for:

  • SMS FAILED — a text message did not deliver. Usually a Twiio issue. Note the error and contact Tech Support if it persists.
  • Chain exhausted — no volunteer accepted the replacement. The slot is still unfilled.
  • Assignment not found — a Manual Replace request used an incorrect assignment. Check and resubmit.
⚠️ If you see repeated errors, do not keep retrying. Note what you were doing and the error message, and contact tech support. Retrying a failed action can sometimes create duplicate assignments.
Part C Overall Coordinator Tasks All rosters · Volunteer management · System settings


Step 1 — Volunteer Sync from Infoodle

Infoodle is a membership database. Rostas syncs volunteer data from Infoodle every morning at 2am automatically. You can also run a manual sync at any time Using the ( Commands view - Pull latest member dater from infoodle).

Run a manual sync when:

  • A new volunteer has been added to Infoodle and needs to appear in Rostas immediately.
  • A volunteer's contact details (email or mobile) have changed in Infoodle.
  • You are about to build a roster and want to make sure you have the latest data.
Step 1 — Sync volunteers from Infoodle

In Rostas → Run Commands, tap 1. Sync volunteers from Infoodle. The sync runs in two steps and takes about 1–2 minutes. The result confirms how many volunteers were updated. Check the Volunteers view to confirm new or updated records appear.

⚠️ Important: Volunteer name changes and Group/Roster membership are only editable in Infoodle, not in Rostas. Always update volunteer details in Infoodle first, then run a sync.


Adding a New Volunteer

New volunteers must be added to Infoodle first and assigned to the correct Infoodle group for their roster type. Once in Infoodle, run a manual sync and they will appear in Rostas. You will then need to set their role preferences before they can be included in a roster build.


Settings and Configuration

System settings control how Rostas behaves — coordinator email addresses, reminder timing, and which features are active. Managed through the Settings Dashboard view in Rostas.

✅ Safe to Change 🔒 Leave to Tech Support
  • Coordinator name and email for each roster type
  • Coordinator mobile number (for SMS test mode)
  • Reminder timing — minutes between SMS replacement messages
  • Email and SMS message wording templates
  • SMS_ENABLED and SMS_TEST_MODE — control live SMS delivery. Contact Guy before changing.
  • WEBAPP_URL — volunteer web app address. Only changes on redeployment.
  • INFOODLE_GROUP_MAP and Infoodle credentials — technical settings for sync.
  • ROSTER_FREEZE_WEEKS — controls which dates are protected from rebuild.
⚠️ If you are not sure whether a setting is safe to change, contact Guy first. Some settings affect the whole system across all roster types. An incorrect value can prevent roster builds or stop emails and SMS from sending.


When to Contact Tech Support

Most day-to-day operations are fully self-contained in Rostas. The situations below require more technical involvement:

  • The volunteer web app URL has changed and needs updating (after a system redeployment).
  • Repeated errors appear in the Status Dashboard that do not resolve on their own.
  • SMS messages stop delivering to volunteers.
  • The system triggers need to be reinstalled (e.g. after a technical issue).
  • Any change to Infoodle connection settings or API credentials.
  • Anything that requires opening the Apps Script editor.

Adding a New Roster Type

New roster types (e.g. a Community Meal) can be added by an Overall Coordinator directly through Rostas without needing to contact Tech Support. The process takes about 5 minutes and the system scaffolds all required Config keys automatically.

⚠️ Before you start: Confirm the Infoodle group for the new roster type exists and you have its numeric Group ID. If the group has not been created yet, create it in Infoodle first, add the relevant volunteers to it, then come back here.


Step 1 — Add the Roster Type in Rostas

Step 1 — Add the Roster Type in Rostas

Go to Rostas → Settings Dashboard → Add New Roster. Fill in the form:

  • Roster Name — the internal name (e.g. Community_Meal). Title case, no spaces in the middle.
  • Coordinator Name, Email, Phone — the coordinator for this roster type.
  • Calendar Time Slot — display order on the Seasons Calendar. Use 3 or 4 for a new type if the Roster is the 3 and 4 roster on that day.  This gives different rosters visual separation in the Calendar.
  • Infoodle Group ID — the numeric group ID. If unsure, run Knox Roster → Developer tools → List all groups.

Save the row. This automatically queues an addRosterType command in the Commands tab.


Step 2 — Wait 1 Minute

Step 2 — Wait 1 Minute

The command poller picks up the addRosterType command and scaffolds all Config keys:

  • COORDINATOR_{TYPE}, COORDINATOR_{TYPE}_EMAIL, COORDINATOR_{TYPE}_PHONE
  • SEASONS_SLOT_{type}, ROSTER_BUILD_WEEKS_{type}
  • Adds the type to ROSTER_TYPES and INFOODLE_GROUP_MAP

Check the Command history panel — the result shows the number of keys added.


Step 3 — Add Roles in Edit Roles

Step 3 — Add Roles

Go to Rostas → Edit Roles. Add each role for the new roster type — set Roster_Type to the new name exactly. Set Is_Active = Active, and set Min/Max People as needed.


Step 4 — Add a Season

Step 4 — Add a Season

Go to Rostas → Seasons Dashboard. Add a new season — set start and end dates, frequency (Weekly or Fortnightly), day of week, and Is_Active = TRUE.


Step 5 — Refresh Calendar and Buttons

Step 5 — Refresh Calendar and Buttons

In Rostas → Run Commands:

  • Run 0. Refresh calendar — rebuilds the Seasons Calendar to include the new roster type's dates.
  • Run 7. Maintenance → Refresh command buttons — new Build, Send, and Preview buttons appear for the new type.


Step 6 — Import Historical Data (if applicable)

Step 6 — Import Historical Data

If you have an existing spreadsheet with past assignments for this roster type, import it now. This gives the fairness engine accurate history from day one so it won't re-roster the same people the week after they were already rostered. See Appendix A1 — Import Existing Roster Data for full instructions. If there is no existing history, skip this step.


Step 7 — Set Preferences and Run

Step 7 — Set Preferences and Run

Set Ministry_Pref for volunteers (via Individual Details or after Volunteer Sync once the Infoodle group is populated). Send role preference emails if the roster type uses specialist roles. Then follow Steps 3, 4, 5, and 6 of the Roster Cycle to collect availability, build, preview, and send.

💡 Refresh command buttons: Run 7. Maintenance → Refresh command buttons any time the Run Commands panel looks wrong — missing buttons, wrong labels, or buttons for inactive types still appearing. This is safe to run at any time.


Removing a Roster Type

If a roster type is no longer needed, it can be removed. This deactivates its roles and seasons, removes its Config keys, and rebuilds the Run Commands buttons. Historical assignment records are not deleted — they remain in RosterAssignments for reference.

⚠️ This cannot be undone automatically. Make sure you genuinely want to remove the roster type. If you might need it again, set its season Is_Active = FALSE instead — that stops it appearing in builds without removing any data.

Coordinator-Triggered Decline and Replace

The standard Manual Replacement (Section 10) lets you pick a specific replacement volunteer directly. Decline and Replace is different — it marks a volunteer as declined and hands control to the automated SMS chain, which contacts available volunteers one by one until someone accepts.

Use this when a volunteer contacts you to say they cannot make it, but you do not have a specific replacement in mind and prefer the system to find one.


Comparison: Manual Replacement vs Decline and Replace

Manual Replacement Decline and Replace 
You choose the replacement?✅ Yes — pick from dropdown❌ No — system finds via SMS

🌙 Quiet hours apply: If you trigger Decline and Replace between 9pm and 7am NZ time, the SMS chain queues until 7am. For urgent same-day slots, contact potential replacements directly in parallel.

D1. Import Existing Roster Data

The Import Existing Roster function loads a manually-made roster into Rostas. It is used in two situations: importing💡 Your text here.📋 Your text here.📋 Your text here. a brand-new roster type from a spreadsheet the coordinator has already been running, or seeding historical data before go-live so the fairness engine has accurate assignment history from day one.

🔒 Access required: This function is only accessible from the Apps Script editor — it is not available in the Rostas AppSheet app. It requires access to the Google account that owns the spreadsheet, and (for AI cleaning) an OpenAI API key stored in Script Properties.


When to Use Import

Use Import when:

  • Setting up Rostas for the first time and you have an existing roster spreadsheet to carry over.  This can help with importing some data (e.g. Role_Type selections and Frequency sellections) providing a smooth transition between the old manually produced roster system and the new Rostas system
  • Adding a new roster type and you want to give the system historical assignment data so it applies fairness rotation from the start.
  • A previous roster season was managed manually and you want it visible in the Roster Assignments history.

Do not use Import to:

  • Fix or edit existing assignments — use Manual Replacement or edit directly in Rostas instead.
  • Re-import data that is already in RosterAssignments — the script skips rows with duplicate Assignment IDs but will add rows if you paste fresh data with new IDs.

Prerequisites

RequirementHow to check
Apps Script accessOpen the spreadsheet → Extensions → Apps Script. You should see the Knox Roster project. If you see a permissions error, you need to be added as an Editor of the spreadsheet.

Step-by-Step: Pivot Format (Recommended)

All steps are run from the Knox Roster menu in the spreadsheet menu bar. Pivot format is recommended when your existing roster is a grid (dates across the top, roles down the side).

Step A — Open the Apps Script Editor

In the Rostas spreadsheet: Extensions → Apps Script. The Knox Roster project opens. You do not need to edit any code. All actions are run from the Knox Roster menu at the top.

Step B — Create the ImportPivot Tab 

If the Import Pivot Tab has not yet been created In the spreadsheet menu: Find the Phase9_ImportRoster Script (left side) click on Knox Roster → Import Existing Roster → Setup — Create Pivot tab. This creates an ImportPivot tab with sample data showing the expected format. Delete the sample rows (rows 2–4) before pasting your data.

Step C — Paste Your Existing Roster

Open your source spreadsheet. Copy the roster grid — including the header row with dates and the role column. Paste it into cell A1 of the ImportPivot tab. Expected layout:

  • Row 1:   Roles  │  13/04/2026  │  27/04/2026  │  …
  • Row 2+: Soup  │  Jane Smith  │  Bronnie H  │  …

Volunteer names can include informal names, initials, and pairs (e.g. "Bruce & Cole"). The AI cleaning step resolves these.

Step D — Run AI Cleaning (Recommended)

In the Phase9_ImportRoster Script menu: → Import Existing Roster → 1. Clean roster with AI. The AI reads your pasted roster and compares every volunteer name and role name against the Volunteers tab and Master_Roles tab. It resolves: nicknames and informal names (e.g. "Bronnie" → "Bronwyn Harrison"), last-name-only references, minor role name variations, and pairs listed in one cell. The cleaned version is written back into ImportPivot. New nickname mappings are saved to the Nickname_Map tab. Review the cleaned ImportPivot tab before importing.

⚠️ If AI cleaning fails: Check that the OPENAI_API_KEY is present in Apps Script → Project Settings → Script Properties. You can proceed without AI cleaning — names that do not match will be flagged in the import result as "Unmatched names". Fix them in ImportPivot and re-run.
Step E — Run the Import

In the spreadsheet menu: Knox Roster → Import Existing Roster → 2. Import (Pivot format). A dialog asks you to confirm the Roster Type. Select the correct type and confirm. The import shows a results dialog:

  • ✅ Written — assignments successfully added to RosterAssignments.
  • ⏭ Skipped — rows already present in RosterAssignments — safe to ignore.
  • ⚠️ Unmatched names — volunteer names that could not be resolved. Cells highlighted red in ImportPivot.
  • ❓ Unknown roles — role names not found in Master_Roles. Add and re-run.

Correct any highlighted cells and re-run until zero unmatched names and zero unknown roles.

Step F — Seed Volunteer Roles from Import

In the Phase9_ImportRoster Script menu → Import Existing Roster → 4. Seed volunteer roles from import. Reads the imported RosterAssignments and creates VolunteerRoles entries for each volunteer-role pairing found. Does not overwrite any existing VolunteerRoles entries.

Step G — Seed Volunteer Frequency from Import

In the Phase9_ImportRoster menu: → Import Existing Roster → 5. Seed volunteer frequency from import. Counts how many times each volunteer appears in the imported data and creates VolunteerFrequency entries. Gives the fairness rotation engine a realistic starting point so the first build does not re-roster the same people who just served.

✅ After steps F and G: The system is ready to build its first roster. Check the Individual Details view for a few volunteers to confirm their frequency counts look right before running the first build.

D2. After a Webapp Redeployment — Sync Preview URLs

Every time the Apps Script web app is redeployed, Google assigns a new WEBAPP_URL. The Rosters tab stores a Preview_URL column that must be updated whenever this changes.

⚠️ Symptoms of a stale URL: The Communications view in Rostas shows broken preview links, or volunteers receive broken links in notification emails. Check WEBAPP_URL in Config first — if it has not been updated since the redeployment, update it before running the sync.
Step 1 — Update WEBAPP_URL in Config

In the Apps Script editor: Deploy → Manage deployments. Copy the new web app URL. In Rostas → Update Settings, find WEBAPP_URL and paste the new URL. Save.

Step 2 — Sync Preview URLs

In the spreadsheet menu: Knox Roster → Developer tools → Sync Roster Preview URLs. Reads WEBAPP_URL and writes the correct ?view=preview&roster=X URL for every active roster in the Rosters tab. Inactive rosters have their Preview_URL cleared automatically.

Step 3 — Confirm

Open WEBAPP_URL?view=preview in a browser. The roster type selector should load. If it shows a permissions error, check the deployment is set to Anyone access.

💡 Also runs automatically: syncRosterPreviewUrls() triggers automatically whenever a roster's Is_Active status is toggled in the Rosters tab, so manual running is only needed after a URL change.

D3. Audit Log Pruning

The Audit_Log tab records every system event. Over time this tab grows large. The pruning function trims it to a manageable size without deleting recent entries.

One-Time Setup

Install the Pruning Trigger

In the Phase9_ImportRoster → Setup — Audit Log Pruning. Installs a weekly time-based trigger that automatically prunes the Audit_Log. Run this once at go-live. Do not run again — duplicate triggers cause double-pruning.

Manual Prune

Prune Now

In the Dashboard Script→ Prune Audit Log Now. Immediately trims the Audit_Log to the most recent entries. Safe to run at any time. Useful if the tab has grown very large before the trigger was installed.

💡 What gets kept: All ERROR-level entries are always retained. INFO entries older than the configured retention window (default 90 days) are removed. The window can be adjusted in Config — search for AUDIT_LOG.

D4. Clean VolunteerRoles Tab

The VolunteerRoles tab can accumulate blank rows or duplicates over time — particularly after bulk imports or if volunteers submit the role preference form more than once. The cleanup function removes these safely.

Run Cleanup

In the spreadsheet menu: Knox Roster → (run from menu) — the function is cleanVolunteerRoles in Maintainance.gs. If it is not in the menu, open the Apps Script editor, select cleanVolunteerRoles from the function dropdown, and click Run. What it removes:

  • Rows with a blank Volunteer_ID or Role_ID.
  • Rows where Role_Type is not Primary or Fillable.
  • Duplicate Volunteer_ID + Role_ID combinations.
✅ Safe to re-run: The function only removes invalid or duplicate rows — it never deletes a valid, unique role assignment. A toast notification shows the count of rows kept and removed.
